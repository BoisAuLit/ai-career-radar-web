## Objective

Create a small **hand-labeled** eval set for G2.1 taxonomy classification
in the pipeline repo:

```
corpus/evals/taxonomy_eval/G2.1/
  ├─ README.md
  ├─ labeling_guide.md
  ├─ eval_set.jsonl        ← ~30 hand-labeled JSON lines
  └─ examples.md            ← optional readable summary table
```

The eval set is the **ground-truth anchor** for the future G2.1d
classifier prompt / scoring dry-run, G2.1e dry-run comparison, and
G2.1f promotion decision. It is the only honest way to measure whether
a prompt rewrite actually does what we want.

**This is pure data / documentation work.** No LLM calls, no classifier
runs, no DB / generated corpus mutation, no runtime behavior change in
either repo. The reference for what "ground truth" means is the spec
shipped in G2.1b: `corpus/taxonomy/G2.1_spec.md` (commit `f833bfd`).

## Implementation outline (for the executor)

### File 1 · `corpus/evals/taxonomy_eval/G2.1/README.md` (~30–50 lines)

- Purpose: ground-truth eval set for G2.1 taxonomy classification.
- Relation to spec: cites `corpus/taxonomy/G2.1_spec.md` (commit
  `f833bfd`) as the labeling reference.
- Statement: hand-labeled; classifier output must NOT be stored as
  truth.
- Usage: read by G2.1d/e dry-run comparison; never by production
  runtime.
- Versioning: this folder is for **G2.1**. Future spec versions get
  their own subfolders (`corpus/evals/taxonomy_eval/G2.2/`, …).

### File 2 · `corpus/evals/taxonomy_eval/G2.1/labeling_guide.md` (~80–120 lines)

- Category-definition summary table (one line per archetype, pointing
  at the canonical definition in `G2.1_spec.md`).
- Labeling rules:
  - How to choose `expected_role_archetype`.
  - How to choose `expected_decision` (`include` / `review` / `exclude`).
  - How to set `expected_rejection_reason` (`null` / `non_ai_swe` /
    `ai_buzzword_only`).
  - How to record `evidence_quotes` (verbatim phrases from JD).
  - How to handle ambiguous examples (default to `review`).
  - How to avoid overfitting labels to the current classifier (label
    what the role *is*, not what the current classifier would predict).

### File 3 · `corpus/evals/taxonomy_eval/G2.1/eval_set.jsonl` (~30 lines)

JSONL — one JSON object per line. Each object has at least:

```jsonc
{
  "eval_id": "G2.1-001",                  // sequential, zero-padded
  "source": "hand_selected",              // or "review_queue_resolved" if drawn from prior queue work
  "company": "Anthropic",                 // human-readable
  "title": "Staff Software Engineer, Android",
  "jd_excerpt": "…short verbatim phrase or paragraph…",   // ≤ ~400 chars
  "expected_decision": "include",          // include | review | exclude
  "expected_role_archetype": "applied_ai", // one of the 11 included, or null when expected_decision != include
  "expected_rejection_reason": null,       // null | "non_ai_swe" | "ai_buzzword_only"
  "expected_axes": {                       // optional scoring-block expectations
    "applied_vs_research": "applied",
    "infra_vs_product": "product",
    "customer_facing": false
  },
  "evidence_quotes": [                     // verbatim phrases from jd_excerpt that justify the label
    "build apps that harness the transformative power of advanced language models"
  ],
  "label_rationale": "one-sentence why a human gave this label",
  "trap_type": "true_positive",            // true_positive | false_positive_trap | false_negative_trap | confusion | reject
  "notes": ""                              // free text; usually empty
}
```

**Target composition** (≥ counts; exact distribution can wobble if it
makes examples more natural):

| trap_type / category                                  | min count |
|-------------------------------------------------------|----------:|
| true_positive · `applied_ai`                          | 5         |
| true_positive · `ai_product_engineering`              | 5         |
| true_positive · `agent_engineering`                   | 5         |
| true_positive · `forward_deployed`                    | 5         |
| false_positive_trap (generic SWE/product/ops with AI buzz) | 5    |
| false_negative_trap (genuine AI roles that look generic) | 5     |

Plus **at least one example each** from these adjacent categories so
the eval set covers the long tail (no minimum; quality > balance):
- `llm_infra`, `eval`, `data_eng_for_ai`, `ml_platform`,
  `research_engineer`, `ai_devrel`, `non_ai_swe`, `ai_buzzword_only`.

**Target total**: ~30 lines. Don't pad above 35; don't drop below 25
without an explicit "why we couldn't reach 30" note in the README.

**Sourcing tips:**
- Many examples can come from prior review-queue resolutions captured
  in pipeline-repo reports `corpus/reports/REVIEW_QUEUE_PACKET_APPLIED_AI_20260624.md`
  and `corpus/reports/G1_3*_*` files. Those JDs already have a human
  judgment we can cite.
- For `ai_product_engineering` true positives: pull from items like
  RQ-32 (Anthropic Safeguards Internal Tooling), RQ-86 (Perplexity
  iOS Comet), RQ-143 (Anthropic UI SWE Claude.ai) — known examples that
  ended up in `other` and motivated the new archetype.
- For false-positive traps: RQ-65 (Cursor Product Quality Engineer —
  already excluded), or generic backend roles at AI companies whose JD
  body has no AI specifics.
- For `ai_buzzword_only`: search any source for "Senior Backend" at
  an AI-branded startup where the JD has ≤ 2 AI mentions and zero in
  responsibilities.
