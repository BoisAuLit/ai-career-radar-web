- `ai_solutions_engineer`

**Not merged (per reviewer):** `data_eng_for_ai` and `eval` stay
separate.

For each of the 13 categories, the spec includes:
- canonical enum value (snake_case)
- human label
- include / reject status
- priority for Bohao target user (⭐⭐⭐ / ⭐⭐ / ⭐ / ✖)
- plain-English meaning
- strong positive signals
- weak signals
- negative signals
- example title patterns
- example JD language
- boundary notes vs the most-likely-confused other categories

### Section 4 — Classification rules
- strong include rules
- strong reject rules
- borderline review rules (route to `review_queue`)
- customer-facing carve-out (FDE bias)
- buzzword-only rejection rule (with the `ai_mentions` counter)

### Section 5 — Scoring fields (proposed for dry-run)

Document the proposed dry-run output shape:
- `ai_relevance_score` (float [0..1])
- `seniority_signal` (junior/mid/senior/staff/principal/unknown)
- `applied_vs_research` (applied/research/mixed/unknown)
- `infra_vs_product` (infra/product/platform/mixed/unknown)
- `customer_facing` (bool)
- `rejection_reason` (`non_ai_swe` | `ai_buzzword_only` | null)
- `ai_mentions.total` (int)
- `ai_mentions.in_responsibilities` (int)

**Crucially, write in the spec:**
- These fields are **proposed for dry-run sidecar output only**.
- They are **NOT to be added to the main schema in G2.1b**.
- **No DB columns added in G2.1b**.

### Section 6 — Reject category policy

State explicitly:
- `non_ai_swe` and `ai_buzzword_only` first appear in **sidecar /
  dry-run output** only.
- They are **NOT promoted into the main schema** until a separate,
  explicitly-approved task post-G2.1f.
- Purpose: auditability + source-quality analysis (knowing *why* a JD
  was rejected, not just *that* it was).

### Section 7 — Evaluation hooks (forward reference to G2.1c)

Describe what the future eval set must contain:
- hand-labeled true-positive examples (≥ 5 per ⭐⭐⭐ category)
- false-positive traps (≥ 5)
- false-negative traps (≥ 5)
- confusion examples (pairs of categories most likely to be mislabeled)
- reject-category examples (≥ 5 split between the two reject buckets)
- evidence-quote check rule (every evidence phrase must appear verbatim in the JD)
