## Objective

Add a small, honest, user-visible **corpus snapshot-date
disclosure** to the served web product so visitors know
the served corpus is a **May 14, 2026 snapshot**, not a
live feed. The disclosure ships via the existing
`WEB_BUNDLE_STATS` helper (new display fields) and
surfaces on 1 high-trust homepage location. No
production data change, no bundle swap, no pipeline
touch.

## Background

P2.0a design memo
(`.agent/design_memos/2026-07-05_P2.0a_data_freshness_corpus_promotion_design.md`)
identified that the served bundle is
`generated_at: 2026-05-14T02:17:04+00:00` (~7.5 weeks
old today) and that the *pipeline* bundle
(`web_bundle_pipeline.json`, generated 2026-06-28)
cannot yet replace it because unique_companies would
regress 35 → 8 and applied_ai would regress 47 → 32 —
a credibility regression. Memo §6 recommends P2.0b as
the first stage: **visible snapshot-date disclosure**
before any data movement, before build-time
`WEB_BUNDLE_STATS` codegen (P2.0c), and long before
any bundle swap.

DECISION for P2.0a
(`.agent/decisions/2026-07-05_run_04_DECISION.md`)
verdict is `approve`, endorsing this sequential plan
P2.0b → g.

`WEB_BUNDLE_STATS` currently exposes 4 primitive
constants (`totalJds`, `appliedAiJds`,
`trackedCompanies`, `evidenceQuotesPerReport`). Its
docstring already declares the source-of-truth
`generated_at` value; we lift that from a
docstring-only comment into an actual displayable
constant so UI code can render it consistently.
