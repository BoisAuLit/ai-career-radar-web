# Regression Fixtures

Synthetic benchmark resume + target-role pairs
used as **stable inputs for future real-report
regression testing** of AI Career Radar. Landed
by AgentOps-3c
(`.agent/decisions/2026-07-12_run_01_DECISION.md`
— pending at authoring time). Consumed by the
still-unbuilt report regression harness
scoped in AgentOps-3d (design) and 3e
(prototype).

## 1. Purpose

The fixtures give the future harness a fixed
set of resume + target-role inputs to feed
into the deployed product (`http://localhost:3000`
in early phases; a specific production URL
only with explicit per-run human approval).
For each fixture the harness will:

- paste the target role into the target
  field,
- paste the resume into the resume field,
- click Generate,
- wait for `stage === "done"`,
- save the generated markdown + screenshots
  + metadata,
- compare against the **structural and
  rubric checks** pinned in AgentOps-3a §7.3
  (13 first-version items) and eventually
  §7.4 (7 future items).

A stable input set is the load-bearing
premise: without it, comparing report
quality across time is meaningless.

## 2. Privacy and safety

- **Every fixture is fully synthetic.**
  No fixture in this directory represents a
  real person.
- **No real names, addresses, phone
  numbers, or emails** appear in any
  fixture.
- **No Bohao (project author) resume
  content** is included anywhere.
- **No private user data** ever enters
  this directory.
- **No fixture contains generated report
  content, screenshots, or LLM output.**
  Fixtures are inputs only.
- Synthetic company names — e.g.
  `Northstar Systems`, `Atlas Retail
  Cloud`, `Meridian DataWorks`,
  `Halcyon Health`, `Cascade Digital`,
  `Beacon Analytics`, `Ridgeline
  Foundry`, `Solstice Labs` — are all
  made up. Any resemblance to actual
  companies is unintended and
  coincidental; no claim about a real
  company is being made.
- Fixtures modelled after archetypes
  (senior backend, full-stack, data
  eng, ML-adjacent, enterprise SWE)
  represent **generic career shapes**,
  not any individual.

## 3. Fixture list

| id | file | persona (source → target) |
|---|---|---|
| A | `benchmark_A_backend_to_applied_ai.md` | Senior backend SWE → Applied AI Engineer at a frontier AI product company |
| B | `benchmark_B_fullstack_to_ai_product.md` | Full-stack product engineer → AI Product Engineer (workflow automation + agentic features) |
| C | `benchmark_C_data_to_llm_infra.md` | Data engineer → LLM Infrastructure Engineer (retrieval / serving / model gateway / eval platform) |
| D | `benchmark_D_ml_adjacent_to_agent_engineer.md` | ML-adjacent SWE → Agent Engineer (reliable tool-using agents for business workflows) |
| E | `benchmark_E_enterprise_to_ai_transition.md` | Traditional enterprise SWE → Software Engineer transitioning into Applied AI / AI Engineer roles |

Each fixture file follows the same 8-field
metadata + 7-heading body structure so a
future harness can parse them uniformly.

## 4. How future harness should use these fixtures

Expected flow (defined here, but NOT
implemented in this AgentOps-3c loop):

1. Load fixture file, parse the
   `## Target role input` and
   `## Resume input` sections.
2. Open `http://localhost:3000` in
   Playwright (headless per
   AgentOps-3a DECISION answer #12).
3. Paste the resume into the résumé
   `<textarea>` (id `#resume`) via
   `page.locator("#resume").fill(...)`.
4. Paste the target text into the
   target `<textarea>` via
   `page.locator("textarea[placeholder*='role']").first().fill(...)`.
5. Click Generate (
   `page.getByRole("button", { name:
   /Generate.*report/i })`).
6. Wait for `stage === "done"` (see
   E2E smoke test 2026-07-08 pattern).
7. Save `report.md`, `report.png`,
   `metadata.json`, `cost.json`,
   `verdict.md` under
   `.agent/regression_runs/<run-id>/<fixture-id>/`
   (path pinned in AgentOps-3a §12
   Required artifacts; NOT created by
   this loop).
8. Compute verdict per AgentOps-3a §7.3
   + §8's 3-tier verdict (green /
   amber / red).
9. Compare against baseline stored per
   AgentOps-3a DECISION answer #7
   (small summaries committed;
   full artifacts external).

**Critical**: the harness must not push a
Vercel deploy or commit runtime code
changes as a side effect of any run. Any
harness failure short-circuits and
requires human review (AgentOps-3a §11
incident-note protocol).

## 5. What these fixtures are not

- **Not reference reports.** The
  fixture files contain resume +
  target inputs only. They do NOT
  contain what a "good" generated
  report looks like verbatim. The
  Expected strengths / gaps sections
  are hint checklists, not
  reference-report text.
- **Not baselines.** No baseline
  report has been generated by this
  loop. Baselines are established by
  the first harness run after 3e
  ships.
- **Not seed data for the corpus.**
  `src/data/web_bundle.json` is
  frozen for a separate reason (P2.0a
  memo). Nothing in
  `.agent/regression_fixtures/`
  leaks into the served corpus.
- **Not an eval prompt set.** The
  eval flow at
  `src/lib/eval-report.ts` uses its
  own judge prompts. These fixtures
  are input to the *generation*
  path, not the eval path.
- **Not a coverage substitute for
  real users.** They cover 5
  representative archetype
  transitions. Real-user reports
  will diverge; that's expected and
  fine — the fixtures exist to
  detect *regressions*, not to
  prove *breadth*.

## 6. Versioning policy

- Each fixture carries `version: 1`
  in its metadata.
- Changing a fixture's semantic
  meaning (target role, persona,
  primary gap theme) requires
  incrementing `version` AND
  invalidating any baseline that
  used the prior version.
- Cosmetic edits (typo fixes,
  formatting) do NOT require a
  version bump.
- Fixture changes only ship
  through explicit **TASK +
  RUN_REPORT + DECISION** loop —
  no drive-by edits.
- Renaming a fixture file is a
  breaking change; treat it as a
  new fixture.

## 7. Future expansion

Kept small on purpose. If v2
expands the suite (e.g. adding a
Data Scientist → Research
Engineer benchmark, or a Product
Manager → AI PM benchmark), each
new fixture is its own
scope-and-approve loop under
AgentOps-3g or similar.
Expansion should be motivated by
observed regression gaps, not by
completeness anxiety.

Do NOT add: fixture files that
overlap heavily with A-E, fixture
files with unrealistic seniority
combinations, or fixture files
containing anything that could
identify a real person. Do NOT
add report outputs or baselines
into this directory.
