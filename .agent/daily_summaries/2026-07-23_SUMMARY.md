### Key finding

- **Immediate failure mechanism identified.**
- Both failed Fixture B runs contained **502 Bad Gateway**.
- Application entered `stage: "error"`.
- Harness waited only for `Copy report` and did not detect error state.
- Early application failure was therefore **masked as a 240-second timeout**.
- **Exact upstream 502 origin remains unresolved.**
- **Quote-integrity wrapper was ruled out** because it runs strictly after report text exists.

### Failure chain

1. Fixture B generation initiated.
2. Browser received **502 Bad Gateway**.
3. App entered `stage: "error"`.
4. Error panel rendered `↻ Retry` and `↺ Start over`.
5. `Copy report` never appeared.
6. Harness did not race success against error state.
7. Harness waited to `HARD_LATENCY_MS = 240000`.
8. Run was recorded as timeout / no report.
9. QI wrapper recorded `blocked_no_report` downstream.

### Conclusions

- Fixture B was **not simply generating slowly** for 240 seconds.
- **Raising the timeout would not fix an early 502.**
- QI integration **did not cause** the failure.
- Exact URL, response layer, request-id, body, and provider origin still need diagnostics.

### Migration semantics

- migration **records already-known truth**
- migration does **not** evaluate quote integrity
- migration does **not** promote a baseline
- migration does **not** modify report content
- migration does **not** imply A/B passed the checker

### Grandfathering exception

- future AgentOps-5e-migrate may modify only:
  - Fixture A current `baseline_metadata.json`
  - Fixture B current `baseline_metadata.json`
- only the new `quote_integrity` block may be added
- existing fields must remain unchanged
- no report, screenshot, verdict, manifest, pointer, or source-run artifact may change
- migration must be reviewable and reversible
- migration requires separate TASK, RUN_REPORT, DECISION, and explicit human GO

### Promotion eligibility

Hard blockers:

- legacy verdict not GREEN
- `blocked` or `error` QI state
- missing report / source / DECISION
- RED under blocking mode

Requires review:

- grandfathered baseline
- RED under `telemetry_only`
- AMBER
- schema drift
- explicit compatibility exception
- unresolved prompt / format issues

Non-blocking warnings:

- R2 terminal-punctuation-only
- appendix entry not cited
- other explicitly non-blocking telemetry findings

### Schema compatibility

- exact `schema_version` match by default
- explicit DECISION allowlist for exceptions
- do **not** infer compatibility from version naming
- retain `checker_commit` for auditability
- `blocking_mode` changes require separate policy DECISION
