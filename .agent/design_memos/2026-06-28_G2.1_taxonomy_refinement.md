### What classification looks like today (essentials)

- Each JD is fetched once → classified once → extracted once (when
  `decision='include'`) → written to `extractions` table.
- `internal_confidence` thresholds in `classifier.py`:
  - ≥ 0.85 → `include`
  - 0.55–0.85 → `review` (goes to `review_queue` for human triage)
  - < 0.55 → `exclude`
- `role_family` is 0–2 archetypes the classifier *suspects* the role
  belongs to, but the **extractor has the final say** (`extractions.role_archetype`
  is the authoritative bucket). This is why G1.3a's extractor overrode
  7/10 of the classifier's archetype guesses.
- There is **no explicit "buzzword-only" reject category** today — such
  jobs just land at `exclude` with confidence < 0.55, and there's no
  audit trail of *why* the classifier rejected them.

### Why G2.1 is being raised now (carry-forward of G1.3 findings)

- The `other` bucket reached 16 items by end-of-G1.3, with 7 of them
  being clearly AI-flavored IC engineering jobs (product/UI/mobile,
  AI-security). G2.0 confirmed the data supports splitting `other`.
- The classifier sometimes accepts AI-buzzword-only jobs (e.g. generic
  Android SWE "uses AI tools" → was provisionally classified `applied_ai`
  before extractor demotion). No auditable reject reason.
- Bohao's user-as-product-owner question is: "do these classifications
  reflect roles I'd actually target, or are they over-inclusive?"

## 2. Target taxonomy

13 categories. **Includes 11 existing + 2 new + 2 explicit reject buckets.**

For each: plain-English meaning, positive/weak/negative signals, example
title patterns, example JD language, whether to include in corpus, and
priority for Bohao-as-target-user.

### Naming convention

`role_archetype` enum (snake_case). Bohao's user-facing taxonomy can use

### Inclusion priority scale

| ⭐⭐⭐ | direct target (Bohao's top funnel) |
| ⭐⭐ | adjacent (worth showing) |
| ⭐ | broader market context (low-prio inclusion) |
| ✖ | reject / out-of-corpus |

### 2.1 `applied_ai` · Applied AI Engineer ⭐⭐⭐

**Meaning.** IC engineer building AI features into a product. Touches
RAG, prompt engineering, fine-tuning data, shipping LLM-powered features
to end users.

**Strong positive signals:**
- Title says "Applied AI", "AI Engineer", "LLM Engineer"
- JD names RAG, prompt eng, vector DB, fine-tuning, evals together with "ship to production"
- Owns a product feature whose value comes from a model call

**Weak signals:**
- Mentions OpenAI/Anthropic/Cohere SDK without describing what's built
- "Builds with AI tools" generically

**Negative signals:**
- "Uses Copilot / Cursor / Claude Code internally for productivity"
- "AI augments our workflow" without the candidate owning the AI parts
- Title is "Software Engineer" with AI in a single bullet of 30

**Example titles:** Applied AI Engineer · LLM Engineer · Senior AI
Engineer · AI Platform Engineer (when product-facing, not infra)

**Example JD language:** *"Build customer-facing RAG pipelines over our
support corpus; own eval harness; ship LLM-driven onboarding flow to
prod."*

**Include in corpus:** yes.

### 2.2 `ai_product_engineering` · AI Product Engineer ⭐⭐⭐ (NEW)

**Meaning.** Frontend / mobile / consumer-UI / internal-tool IC at an
AI-native company. Doesn't write model/agent/eval code, but ships the
product surface that exposes the AI.

**Strong positive signals:**
- Title is Frontend / iOS / Android / UI / Product Engineer
- Employer is an AI-native company (Anthropic, OpenAI, Cursor,
  Perplexity, LangChain, Fireworks, …)
- JD scope is "ship the chat UI / dev portal / mobile client"

**Weak signals:**
- Frontend role at an AI company that also mentions "occasional
  integration with model APIs"

**Negative signals:**
- Frontend role at a non-AI company that lists "AI features" as a side
  project
- Role is `applied_ai` (writes RAG/prompt/agent code) — see boundary in §2 of G2.0

**Example titles:** UI Software Engineer, Claude.ai Consumer Product ·
iOS Software Engineer, Comet · Frontend Engineer, AI Observability &
Evals Platform

**Example JD language:** *"Build the consumer web app for Claude.ai;
focus on streaming render, file upload UX, conversation history; work
with the model team but don't write inference."*

**Include in corpus:** yes. Particularly important for Bohao because his
React + Next.js background is the cleanest on-ramp into AI-product work.

### 2.3 `llm_application_engineer` · LLM Application Engineer ⭐⭐ (NEW, narrow split of applied_ai)

**Decision rule.** If we add this category, `applied_ai` becomes "any
production AI feature work" and `llm_application_engineer` becomes the
narrower "LLM-app-specific" subset. **Open question:** do we actually
need this split, or is the boundary too fuzzy to defend? See §9 risks.

**Recommendation in this memo:** *do NOT add this category in G2.1.*
Keep it conceptually within `applied_ai`. Revisit only if a user-side
report ever needs to filter LLM-app-only roles. Captured here for
completeness because the user's spec listed it.

### 2.7 `ai_data_eval_engineer` · AI Data / Evaluation Engineer ⭐⭐ (CONSOLIDATE of `data_eng_for_ai` + `eval`?)

**Open question.** Current taxonomy has two thin buckets:
`data_eng_for_ai` (training-data pipelines for ML) and `eval` (LLM eval
harnesses, observability). The user's spec proposes combining them.

**Tradeoff:**
- **Combining** simplifies the taxonomy (one fewer bucket) and reflects
  the reality that many JDs blend "build the eval pipeline AND maintain
  the training data".
- **Keeping separate** preserves signal precision — a pure RLHF data
  curator and a pure judge-LLM-runner are quite different roles.

**Recommendation in this memo:** keep `data_eng_for_ai` and `eval`
separate for now. The combined ⭐⭐ priority isn't enough to justify the
churn of merging two existing buckets. Revisit if Bohao's report ever
shows "I keep getting both kinds of roles mixed in the same skill panel
and the recommendations are unhelpful".

If we do split or merge, both files in `corpus/web_bundle*.json` will
need to map the deprecated value to the new one for backward compat.

### 2.13 NEW EXPLICIT REJECT CATEGORIES

#### `non_ai_swe` · Non-AI Software Engineer ✖

**Meaning.** Software engineer at a non-AI company OR at an AI company
but on a clearly non-AI surface (e.g. payroll team at OpenAI). No
meaningful AI engagement in the JD.

**Why surface this explicitly?** Today these are silently
`exclude`-d. Surfacing them gives an auditable count of "how many JDs
the classifier rejected as non-AI" and lets us spot-check whether the
classifier is over-rejecting genuine AI roles.

**Include in corpus:** **no** (filtered out for live product), but
**recorded in `extractions` for audit**.

#### `ai_buzzword_only` · AI-buzzword-only / reject ✖

**Meaning.** JD mentions "AI" or "ML" exactly once in a buzzword
context (often a marketing paragraph) but the actual role and
responsibilities are non-AI. The classic "Senior Backend Engineer at
$startup — we're building the future of AI-powered $thing" where the
backend role is just generic CRUD.

**Why a separate reject category?** To distinguish "rejected because
not AI at all" (`non_ai_swe`) from "rejected because AI was a
buzzword, not a job function" (`ai_buzzword_only`). The latter signals
*source quality* issues — if a source produces many of these, that
source is noisy.

**Include in corpus:** **no**, but **recorded in `extractions` for
source-quality auditing**.

## 3. Classification rules

Proposed rule set the classifier prompt should encode. The rules read
top-to-bottom; first match wins for the strong cases.

### 3.1 Strong include (any one is sufficient)

- **Stack signal:** JD names RAG, fine-tuning, prompt engineering, eval
  harness, vector DB, agent framework, MCP, distributed training,
  inference serving, vLLM/TGI/TensorRT, KV cache, or similar **as a
  candidate responsibility**.
- **Product signal:** JD describes shipping/owning a customer-facing
  feature whose value comes from a model call.
- **Team signal:** JD says "AI team" / "ML platform team" / "agent
  team" / "frontier lab" / "model serving team" as the candidate's home.

### 3.2 Strong reject (any one is sufficient)

- **Title signal:** Title clearly non-engineering (Recruiter, Sales,
  Marketing, Finance, Legal, Comms, Brand, Partnerships, Account
  Executive, Customer Success, Program Manager, EA, Chief of Staff).
  → `non_ai_swe` if at an AI company; reject outright otherwise.
- **Buzzword-only signal:** "AI" / "ML" appears ≤ 2 times in the JD AND
  not in any responsibility bullet. → `ai_buzzword_only`.
- **Internal-AI-tool signal:** JD says "we use AI tools internally"
  (Cursor / Copilot / Claude Code as developer aid) without the
  candidate building or maintaining AI systems. → `non_ai_swe` if
  generic role, else weak signal.

### 3.3 Borderline (route to `review_queue`)

- Title contains "Engineer" + company is AI-native, but JD body has no
  RAG / model / agent specifics.
- JD has hands-on AI language *but* hedged ("primarily X but also Y"
  where X is non-AI).
- Multi-archetype JD where the dominant archetype isn't obvious from
  the first 2 paragraphs.

### 3.4 Customer-facing carve-out

If JD says "customer-facing AI deployment", "embedded with a customer",
or "translate enterprise needs into LLM systems" → bias toward
`forward_deployed` (or `ai_solutions_engineer` if we add it).

### 3.5 Buzzword test (auditable counter)

The classifier should emit an explicit counter:

- `n_ai_mentions_in_jd` — total occurrences of "AI" / "LLM" / "ML" / "GenAI" / model names
- `n_ai_mentions_in_responsibilities` — occurrences specifically inside the responsibility bullets

If both are ≥ 1 and responsibilities is ≥ 2 → not buzzword-only.
If total ≤ 2 and responsibilities = 0 → strong buzzword-only signal.

This makes the buzzword decision a **transparent rule**, not a black-box
LLM judgment.

## 4. Scoring model proposal

Today's `classifications` table has: `is_ai_related` (bool),
`internal_confidence` (float), `decision` (enum),
`role_family_json` (0–2 archetype list), `reason` (string),
`evidence_terms_json` (list of phrases).

**Proposal: add a small structured scoring block, all derivable from the
LLM call's output, all auditable.** Keep the schema small and explicit.

```jsonc
{
  // existing fields stay
  "is_ai_related": true,
  "internal_confidence": 0.82,
  "decision": "include",
  "role_family": ["applied_ai", "agent_engineering"],
  "reason": "Builds RAG over support docs; ships eval harness; LLM-orchestration tool use core to the role.",
  "evidence_terms": ["retrieval-augmented", "eval suite", "tool calling"],

  // NEW · all stay simple, all explainable
  "ai_relevance_score": 0.86,           // bounded [0..1]; informs decision but kept separate from internal_confidence
  "seniority_signal": "senior",         // from JD title + years_experience phrasing; already extracted later but useful at classify time
  "applied_vs_research": "applied",     // "applied" | "research" | "mixed" | "unknown"
  "infra_vs_product": "product",        // "infra" | "product" | "platform" | "mixed" | "unknown"
  "customer_facing": false,             // true only if the role embeds with external customers
  "rejection_reason": null,             // one of "non_ai_swe", "ai_buzzword_only", null
  "ai_mentions": {                      // the auditable buzzword counter from §3.5
    "total": 14,
    "in_responsibilities": 6
  }
}
```

**Field-by-field rationale:**

- `ai_relevance_score` ≠ `internal_confidence`. The latter is "how
  confident am I in my decision", the former is "how AI-relevant is
  this JD". A high-confidence reject still has low relevance.
- `applied_vs_research`, `infra_vs_product`, `customer_facing` — three
  orthogonal axes that help downstream filtering (e.g. Bohao's report
  could say "show me only `applied` × `product` × `customer_facing=false`").
- `rejection_reason` makes reject decisions auditable. Today the
  reasoning is in `reason` (free text); G2.1 promotes the two main
  reject categories to an enum so we can count them.
- `ai_mentions` block: the only counter — kept tiny because the cost
  of expanding it later is low.

**What we deliberately DON'T add:**

- A multi-dimensional "AI engineering style" vector. Premature
  abstraction; would just absorb LLM noise.
- A learned scoring head. We're at ~1500 JDs total; not enough to
  justify any trained component. Stay rule-based.
- Per-skill confidence scores. The `extractions.skills_json` already
  handles skills downstream.

## 6. Evaluation plan

### 6.1 Hand-labeled eval set

Create `corpus/evals/taxonomy_eval/G2.1/` with ~30 JDs:

- **5 true-positive examples per `⭐⭐⭐` category** (`applied_ai`,
  `ai_product_engineering`, `agent_engineering`, `forward_deployed`) →
  20 JDs.
- **5 false-positive traps**: JDs the classifier has historically
  over-included. From G1's review-queue resolutions (Cursor User Ops
  SWE, Scale AI New Grad, etc.).
- **5 false-negative traps**: JDs the classifier has historically
  under-included (the AI-product-engineering items that ended up in
  `other`).

Each entry stores: source URL, full JD text, expected
`role_archetype`, expected `decision`, expected `rejection_reason` (if
any), one-line label rationale.

### 6.2 Metrics (deliberately simple — no learned baseline available)

- **Per-category precision/recall** on the eval set, computed by
  hand-comparing classifier output to expected labels.
- **Confusion matrix** between expected vs predicted `role_archetype`.
  Pretty-printed as a markdown table.
- **Reject coverage**: how often does the new classifier correctly
  rout to `non_ai_swe` or `ai_buzzword_only`?
- **Cost**: estimated $ per 100 classifications (should stay near the
  current ~$0.001 per Haiku call).

### 6.3 Evidence-quote checking

For every `include` decision, the classifier must emit ≥ 1 evidence
phrase (already does). G2.1 adds: the evidence phrase must actually
appear in the JD text (verified by a tiny script).

### 6.4 Manual spot-check process

After the dry-run (G2.1e), Bohao reads:

- The full confusion matrix
- The 5 examples per cell where prediction != expected
- The new reject buckets' top-10 by count

If the spot-check surfaces a systematic miss, iterate the prompt and
re-run G2.1e. Only promote to production (G2.1f) after a clean spot
check.

| # | Risk | Likelihood | Mitigation |
|---|---|---|---|
| 1 | **Over-filter**: G2.1's stricter buzzword reject removes genuinely AI-adjacent roles | Medium | G2.1c eval set explicitly includes 5 false-negative traps; G2.1e dry-run surfaces this before promotion |

## 9. Recommendation

### What to do next

**Approve this memo (G2.1a), then proceed to G2.1b in a separate
TASK.** G2.1b is `yellow` — just a frozen spec doc in the pipeline repo
that downstream tasks reference. Once G2.1b is shipped, decide between:

- **Path A (recommended): G2.1c next** — build the hand-labeled eval set
  before touching prompts. Anchoring the prompt change to a fixed
  ground-truth makes the dry-run interpretable.
- **Path B**: G2.1d next — write the new classifier prompt and let it
  generate against the existing corpus, then build the eval set as a
  reaction. Faster but risks confirmation bias.

### What requires explicit human approval

- G2.1d (prompt rewrite + version bump, even behind a flag).
- G2.1f (promotion to production cron).
- Any modification to `sources.yaml`, `.github/workflows/`, `src/lib/prompts.ts`, or `src/data/web_bundle.json` — these stay out of scope for the whole G2.1 chain.
- Any back-classification of historical rows (G2.1g) — separate decision after G2.1f stabilizes.

### What I'm explicitly NOT recommending

- Don't add `llm_application_engineer` in G2.1 — split inside `applied_ai` is too fuzzy without more data.
- Don't add `ai_solutions_engineer` in G2.1 — boundary against `forward_deployed` needs more JDs to anchor.
- Don't merge `data_eng_for_ai` and `eval` in G2.1 — current separation provides useful signal.
- Don't introduce a learned scoring head — corpus is too small.
- Don't change cost-sensitive parts (model selection, fine-tuning, etc.).
- Don't promote anything to `web_bundle.json` (the V1 frozen bundle) without a separate, explicitly-approved promotion task.
