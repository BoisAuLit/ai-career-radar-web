## 1. Executive verdict

**Verdict: promising but needs targeted
improvement.** The generation stack is
architected honestly — it grounds every
recommendation in real corpus records, refuses
verified classes of hallucination, exposes a
3-judge eval, and holds a conservative
May-2026 snapshot posture. But **the
sample-report page is visibly richer than the
real streamed markdown output** (a colored
7-day plan grid + gap medallions vs. a plain
markdown blob), **evidence quotes are
model-selected from full JD bodies with no
citation-integrity check**, and **there is a
silent-truncation gap in the stream reader**
that will surface as "the report just stopped
mid-sentence" from a user's perspective. None
of those are hero-copy lies — but each is a
credibility papercut. The right next move is
**one narrow report-integrity task** that
makes the real output as trustworthy at a
glance as the sample-report page implies it
will be, before any prompt rewrites or model
swaps.

Bottom line: keep the architecture. Fix the
"real output ≠ what the sample-report page
sells" gap first. Prompts are not the
bottleneck.

The attribution strip already reworked in

## 3. Report structure

The report system prompt
(`src/lib/prompts.ts:82-119`) enforces
exactly 5 sections in this order:

1. `## Target role (1-2 sentences)` —
   includes the archetype tag and a "but
   actually" sentence if the user's title
   misleads.
2. `## What you already have — don't re-learn
   this (3-5 bullets)` — cites % from skill
   profile.
3. `## Your top 5 gaps, ranked (5 numbered
   items)` — each gap must include: skill
   name, % of target-archetype JDs, why it's
   a gap for THIS user, first concrete step,
   one evidence quote with company.
4. `## Skills you might be over-prioritizing
   (0-3 bullets)`.
5. `## Your single highest-leverage next
   action (1 short paragraph)` — ending with
   `"Reassess in [N weeks] after you've
   shipped [X]."`

Plus a final evidence appendix listing
`jd_id + company + title` for cited JDs
(`src/lib/prompts.ts:119`).

The classifier's system prompt (24-47) also
enforces JSON-only output for one small
object.

**Structure is a good backbone.** But note:

- **No client-side enforcement** of the 5
  sections. If the model omits section 4 or
  reorders them, the client
  (`src/app/page.tsx:1284-1286`) silently
  passes the markdown to `<ReactMarkdown>`.
  There is no parser splitting the response
  into "Section 1 / Section 2 / …". Structure
  lives entirely inside the LLM's compliance
  with the system prompt.
- **No stream-completion sentinel.** The
  reader loop
  (`src/app/page.tsx:542-550`) is a `while
  (true)` on `reader.read()`; whenever
  `done === true`, `stage` flips to `"done"`.
  If the stream truncates mid-word — Anthropic
  network wobble, mid-section — the user
  sees "the report just stopped" with no
  indication of whether it was intentional.
  This is the single most fixable-in-one-loop
  issue in this audit.

## 4. Evidence quality

The prompt supplies (per
`src/lib/prompts.ts:139-146`):

```
### Evidence JD: {r.id} — {r.company} — {r.title}
(location: …, seniority: …, min years: …)

{r.body}
```

That is, the model receives **verbatim full JD
bodies** for exactly 5 records, plus aggregate
`top_skills` (top 18 with pcts), seniority
distribution, and top companies for that
archetype. In deep-dive mode
(`src/lib/prompts.ts:50-60`) it also receives a
`COMPANY-SPECIFIC PROFILE` block.

Evidence selection lives at
`src/lib/corpus.ts:167-203` (`pickEvidenceJds`)
and `src/lib/corpus.ts:128-143`
(`pickCompanyEvidence`):

- **Diversity-first**: first pass picks one JD
  per unique company until `n = 5`, then
  relaxes (`src/lib/corpus.ts:187-193`). Good
  — prevents the 5-quote appendix from being
  five variations of Anthropic.
- **Sorted by `canonical_skills.length` desc**
  (`src/lib/corpus.ts:181`) — picks the
  richest-tagged JDs first. Reasonable proxy
  for "quotable" but doesn't correlate with
  which JD will yield the *most relevant
  quote for THIS gap*. There is no
  gap-conditional evidence selection — the
  same 5 JDs feed all 5 gap sections.

**Grounding rules the prompt enforces**
(`src/lib/prompts.ts:65-80`):

- Only cite pcts/counts that appear in the
  supplied skill profile or evidence JDs.
- No trend claims ("rising", "falling",
  "emerging").
- No company-specific claims about internal
  tech stack unless the evidence JD says so.
- Named-company forbidden examples explicitly
  list: `"DeepMind uses JAX"`, `"Anthropic
  engineers ship on AWS"`, `"OpenAI's stack
  is X"`, etc. This is unusually crisp
  guardrail language.
- If a company's evidence is sparse, name that
  fact honestly ("Anthropic has only 1
  llm_infra JD in the corpus, so this report
  leans on the archetype-wide pattern").

**Where evidence can still go wrong:**

- **Quote-attribution integrity is
  model-honor-only.** The prompt asks for
  "One evidence quote from a real JD (give
  the company)"
  (`src/lib/prompts.ts:97-98`) but nothing
  checks the quoted phrase actually appears
  in one of the 5 JD bodies. The
  groundedness judge
  (`src/lib/eval-report.ts:94-118`) checks
  claim classes but is not a
  `substring-in-corpus` verifier. A quiet
  paraphrase — "Anthropic's applied JDs
  emphasize evaluation pipelines" — could
  slip through even the eval, and would slip
  through the report unless the user
  cross-references the appendix.
- **Same 5 JDs for all 5 gaps.** If gap #1 is
  Kubernetes and gap #5 is prompt evals, the
  same 5-JD pool has to support both. The
  prompt doesn't gate reuse.
- **Full JD bodies in-context.** The model
  can quote from anywhere in each body, and
  since the record includes headers/boilerplate
  ("about the team", benefits copy), a
  low-effort quote could be non-signal (e.g.,
  "We value curiosity" from an "About Us"
  block). Prompt discipline mostly prevents
  this but does not enforce it.

## 5. Personalization quality

Adapts to (each supplied to the prompt via
`buildReportUserMessage` in
`src/lib/prompts.ts:123-192`):

- **Resume text** — verbatim, no truncation
  visible in the code path. Sent between
  fenced markers
  (`src/lib/prompts.ts:166-169`).
- **Stated target** — verbatim
  (`src/lib/prompts.ts:171-172`).
- **Archetype classification** — archetype +
  reasoning + company preferences + level
  hint
  (`src/lib/prompts.ts:174-178`).
- **Archetype skill profile** — top 18 skills
  with pcts, seniority distribution, top
  companies
  (`src/lib/prompts.ts:180-185`).
- **Optional company profile** — if the user
  picks a company with ≥3 JDs
  (`src/lib/corpus.ts:91`).
- **5 evidence JDs** — with full bodies.

**Depth is real.** The prompt does receive
everything a peer engineer would need to write
a real gap report — resume, target, archetype
context, per-skill pcts, actual JD bodies.

**Where personalization gets shallow:**

- **Seniority signal is a hint, not a rule.**
  Level hint comes from the classifier
  (`src/lib/prompts.ts:41`,
  `{"level_hint": "<junior|mid|senior|staff|principal|unknown>"}`).
  The report prompt does not have a section
  telling the model how the seniority should
  change the recommendations (e.g., "for a
  staff-level user, weight architecture-scale
  gaps over syntax gaps"). Seniority
  probably ends up as flavor rather than
  structure.
- **Resume "years-experience per skill"
  parsing is left to the model.** The
  classifier prompt hints that "the tool
  works best with at least one job (with
  bullets describing what you did) and a
  list of your technical skills with years
  of experience" (`src/app/page.tsx:136-138`
  `EXAMPLE_RESUME`), but the model is on the
  honor system for actually reading years.
  Gap-ranking based on "how close the user
  already is" (`src/lib/prompts.ts:99`) is
  therefore soft.
- **The "over-prioritizing" section
  (§4)** is trivial to bail on: prompt lets
  the model write "Nothing flagged"
  (`src/lib/prompts.ts:114`). Under real
  usage it will often say that, which
  weakens the "we're going to save you time
  by telling you what NOT to learn" promise
  in the hero framing.

## 6. Actionability

The `specificity rule for recommendations`
block (`src/lib/prompts.ts:101-111`) is the
strongest part of the prompt. It mandates:

- A **named primary tool** ("use X (or Y if
  you prefer Z)" — never "use X or Y").
- A **time bound** ("this week", "in 2
  weeks"; MUST be absolute-from-today, not
  chained off other gaps).
- A **concrete deliverable** (public GitHub
  repo, deployed URL, written eval report,
  demo).
- **Standalone readability** per gap (a
  developer reading only gap #N can start
  today).
- Explicit named picks: `Fly.io`,
  `pgvector`, `LangChain` — with permissible
  parenthetical alternatives.

This block is what saves the report from
generic-LLM-advice failure mode. The
actionability judge
(`src/lib/eval-report.ts:236-255`)
independently flags recommendations that
assume unbridged prerequisite skills or use
made-up tools. So the "action" side has two
layers of defense.

**Actionability gaps:**

- **No calibration to user seniority + time
  budget.** A staff-level user with 10 years
  of backend probably wants a 6-week arc,
  not "this week ship an RAG demo". The
  prompt asks for time bounds but doesn't
  ask for a time-budget-appropriate arc.
- **The single-highest-leverage section
  (`src/lib/prompts.ts:116-118`)** must end
  with `"Reassess in [N weeks] after you've
  shipped [X]."` That is good discipline;
  but "Reassess" is not a re-generation loop
  the product supports today. The reader
  has no in-app way to come back and update
  the gap analysis after shipping. This is
  a growth-loop gap, not an
  audit-blocking gap.
- **`Reassess in [N weeks]`** is contradicted
  by the earlier "Time bounds MUST be
  absolute from today" rule at
  `src/lib/prompts.ts:104`. In practice the
  final paragraph reads "Reassess in 6
  weeks" and the model treats "6 weeks"
  as absolute-from-today. Reasonable.
  Worth watching in reports where the
  final paragraph accidentally
  cross-references gap #1's clock.

## 7. UX risks

Read from `src/app/page.tsx`:

- **Latency framing is honest** — "~60
  seconds", "Streams to your browser",
  two-step loading meter
  (`src/app/page.tsx:1221-1247`) with
  `aria-live="polite"`. The user sees
  progress.
- **Error surface is unusually good.**
  `categorizeError()`
  (`src/app/page.tsx:360-406`) maps six
  error classes (PDF, 429, timeout,
  network, classifier, generator, fallback)
  to friendly `{title, hint}` pairs, with
  the raw error hidden inside `<details>
  Show raw error (for debugging)`. Retry
  button is prominent
  (`src/app/page.tsx:1412-1445`).
- **Privacy is disclosed above the CTA**
  (`src/app/page.tsx:1064-1102`) — the
  `🔒 What happens to my resume?` `<details>`
  panel names "Sent to Anthropic API",
  "Saved on this server: nothing",
  "Anthropic's API does not use input data
  for training by default", "Stateless".
  This is competitively good — most
  résumé-input tools do not disclose this.
- **Cost/time strip is hardcoded, not
  measured** (`src/app/page.tsx:1120-1128`)
  — see §2.
- **Feedback surface** — mailto CTA
  (`src/app/page.tsx:1387-1408`) auto-
  attaches classification + eval scores.
  Reasonable growth loop.
- **Referrer capture** via `?ref=` param
  (`src/app/page.tsx:465-482`), sanitized
  to `[a-zA-Z0-9_\-.]{≤64}` and stored in
  `localStorage["acr:ref"]`. Reasonable.

**UX risks:**

- **Silent stream truncation is invisible to
  the user.** If the connection drops mid-
  section, the report card just says "done"
  and shows the partial markdown. No "we
  may not have finished — retry?" banner.
  This will read as "the AI just gave up
  halfway through" and will bleed trust.
  See §3.
- **Empty PDF from scanned images ships
  silently to the resume field**
  (`src/app/api/extract-pdf/route.ts:1-58` +
  `src/lib/extract-pdf.ts:1-15` +
  `src/app/page.tsx:486-502`). A scanned
  résumé returns near-empty text on a 200,
  the frontend does `setResume(data.text)`
  blindly, the user submits an empty
  résumé and gets a report grounded in
  nothing. No client-side "hmm, that PDF
  extracted 0 chars — try pasting text
  instead?" gate.
- **No inline eval, no post-generation
  quality nudge.** Eval is user-invoked
  only (`src/app/page.tsx:1306-1315`). Users
  who don't press `📊 Eval this report`
  never see the 0.93 / 0.83 / 0.82 numbers
  the hero promises actually apply to
  *their* report. That is a large trust
  gap between the homepage's hardcoded
  hero stats (`src/app/page.tsx:1519-1553`)
  and their personal experience.
- **No streaming for the classifier**
  (`src/app/api/classify/route.ts` blocks
  on `generateText`). The first ~5-10s the
  user waits with only "Classifying your
  target role..." — no partial evidence
  the model is alive. Small papercut but
  consistent across sessions.
- **No rate limiting / abuse control**
  (per the audit's `NOT FOUND`). Not
  audit-blocking; worth flagging for the
  first real launch.

## 8. Output-quality risks

Specific failure modes to name, ranked
roughly by severity:

1. **Silent stream truncation** rendering as
   an incomplete but ostensibly-finished
   report (see §3, §7). **Highest severity
   for trust.**
2. **Uncited (paraphrased) quotes.** The
   prompt asks for real quotes but nothing
   verifies substring-in-JD. A model that
   paraphrases will pass everything but the
   most careful groundedness judge check.
3. **Boilerplate quote leakage.** Full JD
   bodies include "About Us" / benefits
   copy; a low-effort quote could be
   non-signal.
4. **"Nothing flagged" for the over-
   prioritizing section** (§5). Undermines
   the "save you time" framing.
5. **Shallow resume parsing.** The model is
   on the honor system for years-per-skill;
   gap ranking-by-closeness is soft (§5).
6. **Seniority is flavor, not structure**
   (§5).
7. **No time-budget calibration for the
   overall arc** (§6).
8. **Cost/time strip could drift silently
   from reality** (§2, §7).
9. **Scanned PDF ships empty text
   silently** (§7).
10. **Homepage hero-stats
    (0.93 / 0.83 / 0.82)** are aggregate
    across representative reports, not per-
    user (`src/app/page.tsx:1519-1558`).
    Users who never click `📊 Eval` cannot
    verify their own numbers.
11. **No per-gap evidence pool.** Same 5
    JDs for all 5 gaps (§4).
12. **Two-model architecture drift risk.**
    `claude-sonnet-4-6` and
    `claude-haiku-4-5-20251001` are both
    hard-coded string constants
    (`src/app/api/generate-report/route.ts:17`,
    `src/lib/eval-report.ts:8`). If either
    is deprecated silently, the app 500s.
    Not audit-blocking, worth telemetry
    later.

## 10. Default recommendation

**Candidate 1 — Stream-complete sentinel +
"report may be incomplete" banner.**

Justification:

- Fixes the highest-severity output-quality
  risk (#1 in §8) with the smallest
  possible narrow code change.
- Same yellow-loop cadence as P1.7b/c,
  P1.8a/b, P2.0b (`src/app/page.tsx`
  edit, ≤2 files, build + lint + 15/15
  screenshot).
- Reach: **every** user, not just users
  who click `📊 Eval this report`.
