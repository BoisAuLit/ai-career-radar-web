## 1. Problem

Data freshness is now the largest remaining credibility risk
on the live site. Concretely:

- Served bundle (`src/data/web_bundle.json`)'s
  `generated_at` is **2026-05-14T02:17:04Z**. Today is
  **2026-07-05**. The bundle is **~7.5 weeks old**.
- Every visible corpus claim (443 tracked JDs, 35 curated
  companies, 47 applied_ai JDs) traces to that ~7.5-week-old
  snapshot via `WEB_BUNDLE_STATS`.
- The methodology page itself says: *"Daily automated cron is
  enabled and under observation. The live home page still
  serves the manually-curated May 2026 corpus (443 JDs); the
  automated pipeline now re-fetches the corpus daily and
  writes a separate staging bundle previewed at
  /snapshot-pipeline."* — an honest reader thus knows the
  freshness is stale on the surface today.
- A skeptical visitor who reads the methodology page + clicks
  `/snapshot-pipeline` will see two different bundles: the
  served May 2026 snapshot vs. the automated pipeline output.
  Whether that reads as *"transparent, under observation"* or
  as *"stalled, never promoted"* is now genuinely ambiguous
  and worth explicit product decision.

The credibility gap is: the site claims a maintained corpus
via cron ("daily automated") but visibly serves a bundle from
7.5 weeks ago. This is exactly the class of small-number
mismatch that P1.7 + P1.8 spent 4 loops closing on hero /
model / sample-report / methodology. Not fixing it re-opens
the same category of trust risk in a bigger form.

**The trap**: the naive fix — swap
`src/data/web_bundle.json` ← `web_bundle_pipeline.json` — is
NOT a credibility improvement. It is a credibility
REGRESSION. See §5 Option D for why, and §2 / §3 for the
underlying data.

## 3. Source-of-truth map

The pipeline has more data layers than a promotion path can
casually hand-wave. This section lays them out so the memo's
option comparison in §5 is grounded, and so any future TASK
knows exactly which layer it is or isn't allowed to touch.

- **`sources.yaml`** — the *authoritative* company/source
  registry. 8 entries today. Adds a company only if a
  Greenhouse or Ashby board_token exists AND compliance
  metadata (ToS URL, checked_at, allowed_public_use) is
  filled. Red-zone: source expansion is not a copy fix; it
  involves compliance calls.
- **Raw fetched jobs** — per-source per-run API responses,
  stored at `corpus/runs/YYYY-MM-DD_HHMMSS_<source>.json`.
  Per-source snapshots grow at ~1 file per source per day
  when the cron fires (8 files/day). Ephemeral evidence
  layer.
- **Normalized jobs** — canonicalized job records, keyed by
  `canonical_job_id` (source stable ID like the Greenhouse /
  Ashby job ID). Held in `collector.db`.
- **`collector.db`** — SQLite state file. Also committed at
  `corpus/collector.db`; rehydrated from
  `corpus/state/canonical_dump.json` at the start of each
  cron run. Idempotent-by-design: re-runs skip
  already-classified / already-extracted jobs.
- **Snapshots** — human-readable per-run reports written by
  `snapshot.py`. Live at `corpus/runs/**` (JSON) and
  `corpus/reports/**` (Markdown).
- **`corpus/web_bundle_pipeline.json`** — the automation
  output. Cron writes here. **This is what daily 06:00 UTC
  updates.** Contains 519 records / 8 companies / 32
  applied_ai (see §2).
- **`corpus/web_bundle_staging.json`** — an older
  intermediate bundle (2026-05-19). Purpose unclear from
  bare inspection; likely a snapshot from an earlier F2.x
  phase before pipeline stabilized. Not currently consumed
  by web repo. Candidate for cleanup as a follow-up.
- **`corpus/web_bundle.json`** — the *committed* production
  bundle. Byte-copied to web repo's `src/data/web_bundle.json`
  when the web repo was last refreshed (2026-05-14).
- **`src/data/web_bundle.json`** (web repo) — what the
  Next.js build ingests (via `src/lib/corpus.ts` and
  `src/data/lab/**`). The live product's bundle.
- **`src/lib/web-bundle-stats.ts`** (web repo) —
  P1.7b-introduced primitive constants file that
  `WEB_BUNDLE_STATS.*` reads from. Manually kept in sync
  with `src/data/web_bundle.json` via the JSDoc-documented
  refresh procedure.
- **UI copy** (web repo `src/app/**`) — consumes only
  `WEB_BUNDLE_STATS` post-P1.7. Does NOT directly read
  `web_bundle.json` from client code (server-only
  `corpus.ts` reads it; `WEB_BUNDLE_STATS` is what ships to
  client).

**Key promotion decision points on this map**:

- **Cron** touches only `web_bundle_pipeline.json` (pipeline
  repo).
- **F3 promotion** (per workflow header) means: `corpus/web_bundle_pipeline.json`
  → `corpus/web_bundle.json` (still pipeline repo).
- **Product refresh** additionally means: copy `corpus/web_bundle.json`
  → web repo's `src/data/web_bundle.json` **and** refresh
  `src/lib/web-bundle-stats.ts` constants.
- Registry expansion (adding companies to `sources.yaml`) is
  orthogonal to bundle promotion and is a separate red-zone-
  adjacent change.

## 4. Risks

Enumerated so any future promotion loop reads this before
starting.

1. **Stale served bundle (~7.5 weeks)** — biggest current
   trust risk. Severity: **high (product surface)** but
   **low (technical)** — nothing is broken.
2. **`WEB_BUNDLE_STATS` silent drift** — if
   `web_bundle.json` regenerates but the constants aren't
   refreshed, hero/sample-report/methodology all silently
   display wrong numbers. No build-time check exists today.
   Severity: **medium**. Mitigation: build-time codegen (see
   §5 Option B).
3. **Promoting bad pipeline output** — a bad cron run
   (partial fetch failure, classifier hallucination,
   extraction collapse) that lands in
   `web_bundle_pipeline.json` gets served instantly if a
   future F3 swap is done blindly. Severity: **high**.
   Mitigation: hard acceptance gates (§7).
4. **Duplicate jobs** — cross-run deduplication is the
   pipeline's job (per methodology page §5), but a bug
   post-promotion could double-count. Severity: **medium**.
   Mitigation: duplicate-rate gate (§7).
5. **Broken company counts** — the naive swap case in §2.7.
   Any promotion that drops company count below 35 is a
   credibility regression, not improvement. Severity:
   **high**. Mitigation: `unique_companies ≥ 35` gate (§7),
   or hybrid approach that preserves the served set as a
   superset.
6. **Classifier / archetype quality drift** — if the
   classifier prompt changes (G2.1d red-zone territory)
   between the served bundle's classification pass and the
   pipeline bundle's, archetype counts shift not because
   the underlying data changed but because the model's
   judgment changed. Severity: **medium**. Mitigation:
   `CLASSIFIER_VERSION` stamp + gate that promoted bundle
   is same-version as prior.
7. **Compliance / attribution** — every source in
   `sources.yaml` has `allowed_public_use:
   snippet_with_attribution`. If any source expansion in
   the future adds a company whose compliance metadata is
   absent or in a stricter state, the promoted bundle
   silently violates the site's public-display promise.
   Severity: **high (legal)**. Mitigation: compliance-check
   gate (§7).
8. **Deploy rollback complexity** — Vercel deploys on Git
   push. If a bad bundle lands on `main`, Vercel deploys
   it. Revert requires a Git revert commit (not just a
   Vercel dashboard action) because Vercel serves the last
   built commit. Severity: **medium**. Mitigation: explicit
   rollback plan (§8).
9. **UI confusion with too many dates/statuses** — if we
   surface `generated_at` alongside "manually-curated May
   2026 corpus" and "staging pipeline bundle at
   /snapshot-pipeline", a visitor sees three date/status
   claims and doesn't know which one is real. Severity:
   **medium**. Mitigation: consolidate to *one*
   authoritative "corpus last updated" claim, plus optional
   per-page disclosures.
10. **Pipeline last commit is 7 days ago** — either cron
    stopped, saw no delta worth committing, or GitHub
    Actions is throttling. Severity: **low (memo blocker
    zero)** but **worth verifying** before any promotion
    TASK.

## 5. Promotion path options

Compared honestly with expected value / risk / files touched
/ approval gate / green-yellow-red / why-now-why-not-now.

### Option A — Do nothing except add a visible snapshot-date disclosure

**What it is**: No pipeline touch. Add a small "corpus last
updated: 2026-05-14" surface to homepage (or footer) and
optionally cross-link to `/snapshot-pipeline`. Ship it. Costs
nothing on the data side.

- **Value**: **HIGH** for a small change. Directly addresses
  the largest remaining trust risk (§1) by making the
  freshness *visible and honest*. A visitor no longer has to
  hunt through the methodology page to find the date.
- **Risk**: **low**. Touches 1-2 UI files at most.
- **Files touched**: `src/app/page.tsx` (or a new small
  component under `src/components/`), possibly
  `src/lib/web-bundle-stats.ts` if we want to expose
  `generated_at` alongside the counts. Yellow small copy
  change.
- **Validation**: `npm run build`, `npm run lint`, visual
  check that formatted date renders correctly.
- **Human approval**: yes (Vercel visible deploy).
- **Green/Yellow/Red**: **Yellow** (`.agent/`+`src/` copy).
- **Why now**: Directly closes the strategic-review's top
  freshness risk. Costs one focused loop. Doesn't require
  any pipeline decision or bundle swap. Sets up cleanly for
  Options B-E later.
- **Why not now**: n/a. This is the safest first move.

### Option B — Build-time or script-based `WEB_BUNDLE_STATS` codegen

**What it is**: A tiny Node script (or Next.js build step) at
`scripts/gen-web-bundle-stats.mjs` that reads
`src/data/web_bundle.json` at build time and rewrites
`src/lib/web-bundle-stats.ts` with the correct constants.
Optionally: assert at build time that `web-bundle-stats.ts`
matches the bundle (drift check) — fail the build if the
constants have drifted.

- **Value**: **Medium**. Closes the drift risk (§4 #2)
  before it bites. Currently no consumer has drifted, but 4
  do consume the helper.
- **Risk**: **low**. Small script, easy to review, no
  runtime code path change.
- **Files touched**: new `scripts/gen-web-bundle-stats.mjs`,
  `src/lib/web-bundle-stats.ts` (regenerated), maybe
  `package.json` (script entry) — the `package.json` touch
  makes this **not** trivially green.
- **Validation**: script produces same file on the
  unchanged bundle (idempotent); modifying bundle changes
  file predictably; `npm run build` still passes.
- **Human approval**: yes.
- **Green/Yellow/Red**: **Yellow** (touches
  `package.json`).
- **Why now**: No visible symptom yet.
- **Why not now**: Deferrable. Do it AFTER a bundle
  refresh actually happens; the drift risk becomes real
  only when the bundle actually changes.

### Option C — Preview refreshed pipeline bundle at `/snapshot-pipeline`, no production swap

**What it is**: The `/snapshot-pipeline` route already
consumes a bundle's `generated_at`. Verify that the
`/snapshot-pipeline` page today reads from a fresh
`web_bundle_pipeline.json` copy (not the stale `web_bundle.json`),
and if it doesn't, make it read that pipeline bundle in a
read-only, staging-only surface. **No swap.** No changes to
what the homepage / sample-report / methodology serve.

- **Value**: **Medium**. Makes the "under observation"
  automation credible — a visitor can click through and see
  the 8-company / 519-record pipeline output as of
  2026-06-28, side-by-side with the methodology's stated
  positioning.
- **Risk**: **low-medium**. Touches web repo consumer of
  `web_bundle_pipeline.json`. Does NOT touch pipeline repo,
  does NOT touch the served bundle.
- **Files touched**: `src/app/snapshot-pipeline/page.tsx`,
  possibly `src/data/web_bundle_pipeline.json` (staged
  copy), possibly `src/lib/corpus.ts` if a second reader
  helper is needed.
- **Validation**: `npm run build`; visual check of the
  staging surface; no diff on the served bundle.
- **Human approval**: yes.
- **Green/Yellow/Red**: **Yellow**.
- **Why now**: Useful mid-step between Option A (visible
  disclosure) and later options (D/E).
- **Why not now**: Slightly larger scope than Option A; and
  the exact 8-company-only pipeline output *without any
  editorial framing* may look worse than the served
  bundle (less impressive breadth). Ship this AFTER Option
  A so the visitor already has the "35 companies as of
  May 14" anchor.

### Option D — Promote refreshed bundle to production (full F3 swap)

**What it is**: `web_bundle_pipeline.json` →
`web_bundle.json` in pipeline repo, then copy into web
repo's `src/data/web_bundle.json`, refresh
`WEB_BUNDLE_STATS`, deploy.

- **Value**: **NEGATIVE with current pipeline bundle**.
  Yes, freshness improves (+45 days). But:
  - Unique companies drops **35 → 8** (§2.7). Homepage
    reads "8 curated companies". Methodology reads "8
    companies is a curated sample, not a census". This
    reads as a fire sale.
  - `applied_ai` drops **47 → 32**. Hero says "Based on 32
    applied_ai JDs" — smaller sample, less impressive.
  - The whole P1.7 + P1.8 credibility argument was that
    grounded-in-real-data > glossy claim. Silently
    shrinking the corpus after just documenting it is the
    opposite of that.
- **Risk**: **high**. Undoes credibility work. Unless the
  drop is *disclosed*, this is a straight regression.
- **Files touched**: pipeline `corpus/web_bundle.json`,
  web `src/data/web_bundle.json`,
  web `src/lib/web-bundle-stats.ts`. Red-zone-adjacent.
- **Validation**: acceptance gates (§7).
- **Human approval**: yes, red-zone.
- **Green/Yellow/Red**: **Red** in current state
  (regresses credibility surface). Would become **Yellow**
  only if paired with an expanded `sources.yaml` registry
  that closes the 35 → 8 gap OR a clear editorial framing
  ("automated pipeline covering N of the M curated
  companies", side-by-side).
- **Why now**: **Not now.** Fails §7 gates today.
- **Why not now**: Registry gap (35 curated vs. 8
  automated) makes a naive swap a credibility regression.
  Fix registry gap first (P2.0f or similar future TASK),
  or ship editorial-framed side-by-side (Option C+).

### Option E — Fully automate daily production bundle promotion

**What it is**: Modify the pipeline cron to promote
`web_bundle_pipeline.json` → `web_bundle.json` at the end
of every successful run, then commit both. Vercel
auto-deploys. No human in the loop between cron and
production data.

- **Value**: **HIGH long-term** if pipeline output is
  actually trustworthy; freshness stays ≤24h.
- **Risk**: **VERY HIGH.** BLK-0002 blocks full automation.
  Automates a red-adjacent step (production data swap).
  If a bad run lands, it's live for up to 24 hours before
  the next run.
- **Files touched**: pipeline `.github/workflows/**`,
  pipeline `scripts/collector/**` (promotion step).
- **Validation**: same gates as Option D, but automated,
  which is exactly what Q10 / BLK-0002 says is not OK yet.
- **Human approval**: yes, red.
- **Green/Yellow/Red**: **RED**. Fails BLK-0002 (full
  automation activation) and Q10 (automation-infra pause).
- **Why now**: **NOT NOW**. Blocked by policy.
- **Why not now**: BLK-0002 open. Q10 pause continues.
  Even the AgentOps-2c Shape B prototype (which is *not*
  approved yet) is more conservative than this.

## 6. Recommended staged plan

**Opinionated ordering** — do not run multiple in parallel.

- **P2.0b · Visible corpus snapshot-date disclosure** (next
  code loop). Small yellow UI change: add a `<span>` or
  small badge to homepage (or footer / methodology "Current
  pipeline phase" strip) rendering `WEB_BUNDLE_STATS.generatedAt`
  (extending the helper to expose a date string) or an
  equivalent constant. Optional: cross-link to
  `/snapshot-pipeline` for the "staging output" view. Costs
  1 loop. Directly closes strategic-review's top freshness
  risk without touching pipeline / data / prompts / models.

- **P2.0c · Build-time or script-based
  `WEB_BUNDLE_STATS` codegen + drift check** (loop after
  P2.0b). Small yellow addition of a `scripts/gen-web-bundle-stats.mjs`.
  Runs at build time (or via `npm run gen:stats`). Fails
  the build if the checked-in constants file is out of
  sync with the served bundle. Costs 1 loop.

- **P2.0d · Preview refreshed pipeline bundle audit**
  (loop after P2.0c). Read-only audit of the current
  `web_bundle_pipeline.json` (in the pipeline repo)
  against the served bundle: field-by-field diff, gate
  each acceptance criterion from §7 individually, produce
  a written verdict. **No swap.** Costs 1 loop.

- **P2.0e · Corpus promotion checklist + rollback plan
  hardening** (loop after P2.0d). Formalize §7 gates into
  an executable-adjacent checklist (`.agent/design_memos/`
  or a `docs/PROMOTION_CHECKLIST.md`-equivalent under
  `.agent/`). Explicit rollback plan. Costs 1 loop.

- **P2.0f · sources.yaml registry expansion design memo**
  (loop after P2.0e). *Separate design memo* to scope
  closing the 35 → 8 registry gap. Options: add ATS
  integrations for the ~27 companies covered by the
  manually-curated May 2026 bundle but missing from the
  automated pipeline; or explicitly serve a hybrid /
  side-by-side that discloses the automation coverage.
  This one is yellow-to-red because it touches
  `sources.yaml` (compliance) and possibly
  `scripts/collector/**`. Design-only for the first pass.
  Costs 1 loop.

- **P2.0g · Production bundle swap** (only if P2.0d + P2.0e
  + P2.0f converge on a viable path). Executes the swap
  under the checklist from P2.0e. This is a red-adjacent
  loop and requires explicit human GO — including a
  DECISION from Bohao + ChatGPT after reviewing every gate
  in §7 with real bundle data.

**Do NOT skip stages.** Each stage produces a written
artifact (UI diff, script, audit report, checklist, memo)
that the next stage cites. Do NOT collapse "add codegen
+ swap bundle" into one loop; the codegen deserves its own
DECISION even though it's small.

## 7. Acceptance criteria for any future corpus promotion

Any future TASK that proposes swapping the served bundle
MUST demonstrate ALL of the following before pushing:

1. **`n_records` threshold**: promoted `n_records` ≥ 400.
   Rationale: current is 443; a substantial drop suggests a
   fetch failure and should abort promotion.
2. **`unique_companies` threshold**: promoted
   `unique_companies` ≥ 35. Rationale: current baseline.
   **This gate today FAILS the pipeline bundle** (only 8
   companies) — the whole registry-expansion work in
   P2.0f is what unblocks this gate.
3. **`applied_ai` count sanity**: promoted `applied_ai` ≥
   40. Rationale: current is 47; a drop below 40 risks the
   hero's "47 applied_ai JDs" chip going backwards.
4. **Duplicate rate**: unique
   `record.canonical_job_id` count == `n_records` (no
   dupes). Verified via a small Python script or SQL
   against `collector.db`.
5. **Empty field checks**: 0 records with empty `company`
   or empty `archetype`. Verified via Python one-liner.
6. **Attribution / compliance check**: every source
   contributing records must have
   `allowed_public_use in {snippet_with_attribution,
   full_public_display}` in `sources.yaml`. No `unknown`,
   no expired ToS-check timestamps.
7. **Archetype distribution sanity**: `applied_ai` /
   `agent_engineering` / `llm_infra` / `research_engineer`
   / `ml_engineer` each have at least 5 JDs. Prevents a
   pipeline bundle where one archetype has collapsed to
   near-zero from telling the wrong story on the hero.
8. **Sample-page render**: `/sample-report`, `/methodology`,
   `/` all render with new bundle without any React or
   TypeScript error, with the substituted counts appearing
   in the DOM (spot-check via `npm run build` output

## 8. Rollback plan

**If a promoted bundle is bad and lands in production**:

1. **Detect fast**. Watch these signals in the first
   10 minutes post-deploy:
   - Vercel deploy log completed successfully (build
     passed).
   - Manually load `/`, `/sample-report`, `/methodology`.
     Confirm the visible numbers match the promotion's
     expected values.
   - Bohao or ChatGPT spots-check that the served copy
     doesn't say something obviously wrong (e.g. "8
     companies" if the gate `unique_companies ≥ 35`
     accidentally slipped).

2. **Revert commit sequence**:
   ```
   cd /Users/bohaoli/Desktop/ai-career-radar-web
   git status                      # confirm clean
   git log --oneline -5            # find the bad promotion commit
   git revert <bad_commit_hash>    # creates a revert commit
   git push origin main            # Vercel auto-deploys the revert
   ```
   Do NOT `git reset --hard` on a pushed commit — the
   revert commit preserves the audit trail.

3. **Vercel behavior**: Vercel serves whatever the last
   built `main` commit is. The revert commit triggers a
   fresh build with the *old* bundle contents restored.
   Rollback window is bounded by the Vercel build queue
   (typically 30-120 seconds).

4. **`src/lib/web-bundle-stats.ts` refresh** must also be
   part of the revert. If the promotion commit updated
   the constants alongside the bundle, the revert commit
   will unwind both. Verify.

5. **Root cause + follow-up**: the promotion that failed
   its gates in production means gate coverage was
   insufficient. Add the missed gate to §7 for the next
   attempt. Do not immediately re-promote.

6. **No revert of pipeline repo**: this rollback plan is
   web-repo only. The pipeline repo's
   `web_bundle_pipeline.json` and `web_bundle.json` are
   separate concerns from what the web product serves.
